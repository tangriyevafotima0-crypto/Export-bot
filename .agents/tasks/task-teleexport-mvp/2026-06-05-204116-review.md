# TeleExport desktop app for Telegram chat export (v2)

The four issues from v1 that were flagged as fixable have been addressed: API credentials now flow from the PhoneInput UI through the store to the Python handler, settings persist to `CONFIG_DIR/settings.json`, the PDF ghost option is removed from FormatSelector, and concurrent exports use a dict keyed by `export_id`. All four fixes are correct and complete.

**Watch for:** API credentials entered during login are not persisted, so `check_session` can never reconstruct the Telegram client on a cold start — session resume is broken and the user must re-authenticate every time the app launches (confirmed). Silent failure on settings write means the user gets `{"success": True}` even when disk is full (confirmed, low severity).

**Verdict**: NEEDS_CHANGES

## High-level view

The auth fix correctly wires `api_id`/`api_hash` from UI to backend, but introduces a new gap: on app relaunch, `TeleExportClient.client` is `None` because `init()` was never called with credentials. `check_session` short-circuits with `has_session: False` and the user must re-login every time. The fix is to persist `api_id`/`api_hash` in settings (they're app identifiers, not secrets) and call `init()` before `check_session`.

The settings persistence and concurrent export fixes are both clean. The `_save_settings` swallows `OSError` without propagating failure, but for an MVP this is acceptable.

## Client initialization gap on session resume

`TeleExportClient.__init__` sets `self.client = None`. The only path that calls `init(api_id, api_hash)` to construct the underlying `TelegramClient` is `AuthManager.send_code`. The `check_session` path does:

```python
async def check_session(self) -> dict:
    if self.client.client is None:
        return {"has_session": False}
```

On a cold start after a successful prior login, the `.session` file exists on disk but `client.client` is `None`. The method returns `has_session: False` without ever attempting to load the session. The user is forced to re-enter credentials and re-authenticate every launch.

The fix: persist `api_id` and `api_hash` in the settings file (they are per-app identifiers from my.telegram.org, not user secrets). On process start, if settings contain these values, call `client.init(api_id, api_hash)` before any `check_session` call. This allows the Telethon client to load the existing session file and report authorization status correctly.

## Acknowledged MVP trade-offs (not re-raised)

The following were flagged in v1 and explicitly deferred by design:

- Session files unencrypted at rest (Telethon default)
- Unbounded message accumulation (streaming for v1.5)
- Progress calculation averaging (UX polish)
