# TeleExport: Electron + Python desktop app for Telegram chat export

A complete desktop application that exports Telegram chat histories via a Python/Telethon backend (JSON-RPC over stdin/stdout) wired to an Electron shell with a React/Zustand frontend. The architecture splits concerns cleanly: Python handles Telegram API interaction, export orchestration, and formatting (HTML/JSON/CSV), while Electron manages the process lifecycle and the React layer drives a multi-step auth flow, chat dashboard, and export wizard. The implementation covers 103 new files across both stacks.

**Watch for:** API credentials (`api_id`/`api_hash`) are required by the backend's `auth.send_code` handler but the frontend's `sendCode` action never sends them — the auth flow will fail at runtime (confirmed). Settings are stored only in memory and lost on restart (confirmed). The export engine accumulates all messages in RAM before formatting, which will OOM on large chats (likely). Session files on disk contain full Telegram auth tokens with no encryption at rest (confirmed).

## High-level view

The IPC boundary is a JSON-RPC protocol over stdin/stdout with a request/response pattern plus server-pushed events. The Python side reads lines from stdin, dispatches to registered async handlers, and emits progress events back. The Electron `PythonBridge` spawns the Python process, assigns numeric IDs, and maps responses back to pending promises with a 30-second timeout. This is a clean and appropriate design for an Electron+Python stack.

The auth flow implements Telegram's multi-step login (phone → code → optional 2FA password) with Telethon. The Python handler expects `api_id` and `api_hash` as parameters on `auth.send_code`, but the React auth store only sends `{ phone }` — a contract mismatch that will produce a `KeyError` at runtime. Beyond that, session tokens persist as bare Telethon `.session` files in `~/.teleexport/sessions/` with no additional protection.

Export is the core feature: the engine iterates messages using a Telegram takeout session (with fallback to standard iteration), downloads media with dedup, and then formats the full message list. The "format after collect" approach means the entire message history must fit in memory. Cancellation is cooperative via a `self.cancelled` flag checked between batches but not during media downloads.

The frontend is a conventional React + Zustand stack with typed IPC calls. It compiles and builds cleanly. The `FormatSelector` component offers a "PDF" option that has no corresponding backend formatter — selecting it would produce a KeyError in `_get_formatter`.

## Issues (7)

<details>
<summary>Issues (7)</summary>

1. **Missing `api_id`/`api_hash` in frontend `sendCode`** — The Python `auth.send_code` handler requires `params["api_id"]` and `params["api_hash"]`, but the auth store only sends `{ phone }`. Add these params to the frontend call (prompt the user or store in settings/config).
2. **In-memory-only settings** — `_settings` in `handlers.py` is a plain dict that resets every time the Python process restarts. Persist to a JSON file in `CONFIG_DIR`.
3. **Unbounded message accumulation** — `ExportEngine._export_chat` appends all messages to a list before formatting. For chats with millions of messages, this will exhaust memory. Stream messages to the formatter incrementally.
4. **PDF format option with no backend** — `FormatSelector` lists "pdf" as a format, but `ExportEngine._get_formatter` only maps html/json/csv. Selecting PDF will raise a `KeyError` or silently fall back to HTML (the `dict.get` returns `HTMLFormatter` as default — so it silently formats as HTML, which is misleading).
5. **Session files unencrypted at rest** — Telethon `.session` files in `~/.teleexport/sessions/` contain auth keys in plaintext. Any process with filesystem access can steal the session. Consider OS keychain integration or encrypting at rest.
6. **No `export.complete` event handling for overall progress** — The frontend listens for `export.complete` and calls `setCompleted`, but the overall percent jumps from whatever the last chat's progress was to 100%. The average-of-percentages approach in `updateProgress` gives misleading numbers for chats of different sizes.
7. **Concurrent export race** — `app.current_export` is a single slot. Starting a second export while one is running overwrites the reference, making the first export uncancellable and its `finally` block nulling out the new engine.

## IPC contract mismatch on auth flow

The Python `auth_send_code` handler destructures three required fields:

```python
async def auth_send_code(params, msg_id):
    phone = params["phone"]
    api_id = params["api_id"]
    api_hash = params["api_hash"]
```

The frontend auth store sends only `{ phone }`:

```typescript
const result = await window.teleexport.python.call('auth.send_code', { phone }) as { ... };
```

This will raise a `KeyError: 'api_id'` on every login attempt. The design needs to decide where API credentials live — either the user enters them (add UI fields), or they're stored in a config file the backend reads, or they're bundled (not recommended for open-source).

## Session credential exposure

Telethon session files are SQLite databases containing the user's auth key and data-center configuration. These files sit in `~/.teleexport/sessions/` with standard file permissions. Any local process or malware with read access to the home directory can copy these files and impersonate the user on Telegram without needing their password or 2FA. This is the standard Telethon behavior, but a desktop app distributed to end users should provide at least OS-keychain-backed encryption or restrictive file permissions.

## Memory-bound export for large chats

In `ExportEngine._export_chat`, every message from the fetcher is appended to a list:

```python
messages = []
async for batch in fetcher.iter_batches():
    for msg in batch:
        ...
        messages.append(msg)
formatter.format(chat_name, entity, messages, media_dir)
```

A Telegram channel with 500K+ messages will hold all `Message` objects (each with media metadata, text, sender info) in memory simultaneously. The formatters (`HTMLFormatter`, `JSONFormatter`, `CSVFormatter`) all accept the full list and iterate it — they could instead accept an iterable and write incrementally. The CSV formatter already writes row by row, so streaming would be straightforward there; HTML would need chunked template rendering.

## PDF format ghost option

The `FormatSelector` component declares four options:

```typescript
const formats = [
  { id: 'html', ... },
  { id: 'json', ... },
  { id: 'csv', ... },
  { id: 'pdf', label: 'PDF', description: 'Printable document format' },
];
```

The backend's formatter lookup defaults to `HTMLFormatter`:

```python
formats = {"html": HTMLFormatter, "json": JSONFormatter, "csv": CSVFormatter}
formatter_class = formats.get(self.config.format, HTMLFormatter)
```

So selecting PDF silently produces an HTML file. The user sees "PDF" in the UI but gets HTML output with no warning. Either remove the PDF option from the frontend or implement a PDF formatter (e.g., wrapping the HTML output with a headless Chrome PDF render or weasyprint).

## Concurrent export safety

`app.current_export` is assigned in `export_start` and cleared in the `run_export` finally block:

```python
app.current_export = engine
asyncio.create_task(run_export())
# ... if called again before completion:
app.current_export = engine2  # first engine now unreachable for cancel
```

The cancel handler only looks at `app.current_export`, so the first export becomes orphaned. A dictionary keyed by `export_id` would fix this:

```python
app.exports[export_id] = engine
# cancel:
app.exports.get(export_id)?.cancel()
```

## Settings volatility

The `_settings` dictionary in `register_handlers` is initialized from `DEFAULT_SETTINGS` on every process start. `settings.set` updates the dict in memory, but nothing persists it to disk. When Electron kills and restarts the Python process (which happens on app restart, crash recovery, etc.), all user settings revert to defaults. The `CONFIG_DIR` path is already defined — writing a `settings.json` there on each `settings.set` call would fix this.

## Test coverage shape

68 Python tests cover the auth flow, export engine config, IPC server message handling, formatter output, and media downloader deduplication. Coverage is reasonable for a first pass. Notable gaps: no tests for the actual `run_export` task lifecycle (cancel mid-export, error propagation), no integration test for the full IPC roundtrip (send JSON line on stdin, read response on stdout), and no frontend tests (the `tests/frontend/` directory is empty with only a `.gitkeep`). The `useExportProgress` hook's event subscription logic and the Zustand stores' state transitions are untested.
