# Anti-Stalker Intelligence System

A 13,000-line Telegram surveillance system combining a Telethon userbot, python-telegram-bot command interface, Flask tracking-link server, FastAPI analytics dashboard, APScheduler-driven monitoring loop, and a local ML scoring pipeline (scikit-learn IsolationForest + weighted feature vectors). The system monitors a user's Telegram contacts for "stalking" indicators — story view frequency, online time correlation, group interaction patterns — then scores, profiles, and alerts through both a bot chat and a web dashboard with real-time WebSocket updates.

**Watch for:** Dashboard secret key doubles as JWT signing key and login password with an insecure default (confirmed). Flask trap server and dashboard bind 0.0.0.0 with no TLS (confirmed). Database session usage pattern (`async for session in get_session()`) may produce detached ORM objects accessed after session close (confirmed). Shutdown orchestration in `main.py` references `dashboard_task` that can be unbound if the try block fails early (confirmed). Docker port mapping disagrees with config defaults (confirmed).

## High-level view

The system runs five processes in one Python event loop: a Telethon userbot for Telegram API monitoring, a python-telegram-bot poller for the command interface, a Flask HTTP server (run in a thread) serving tracking links with embedded fingerprinting JS, a FastAPI/uvicorn server exposing the analytics dashboard, and an APScheduler driving periodic analysis jobs. This single-process design means any unhandled exception or OOM in one component takes down all five.

Authentication is owner-only for the bot (MY_TELEGRAM_ID check) and a single shared password for the dashboard (the `dashboard_secret_key` setting). The same secret is reused as both the JWT signing key and the password — meaning anyone who sees a JWT can derive the secret, and the default value is the literal string `"change-me-to-a-random-secret"`.

The ML pipeline operates entirely offline using numpy/sklearn. An 8-feature weighted scorer produces a 0-100 "suspicion score" for each tracked user. IsolationForest anomaly detection runs on 28-day sliding windows. Pattern detection identifies four archetypes (NIGHT_STALKER, IMMEDIATE_RESPONDER, DAILY_CHECKER, SILENT_OBSERVER). These all read from the same SQLite database through async sessions.

The trapnet module deploys honeypot tracking links that serve a fingerprinting page (canvas hash, WebGL, screen dims, timezone) before redirecting. Visitor data is stored and correlated back to tracked users. The trap server uses a mixed sync/async pattern to bridge Flask's sync handlers with the async database layer.

1. **Dashboard secret = JWT key = login password** — Separate the login password from the signing secret. The `dashboard_secret_key` is used as the login comparison value *and* the HS256 JWT key, collapsing two distinct security properties.
2. **Insecure default secret ships in config** — The default `"change-me-to-a-random-secret"` is accepted at startup without warning. Add a startup check that refuses to run with the default value.
3. **No TLS on dashboard or trap server** — Both bind 0.0.0.0 with plaintext HTTP. The dashboard serves JWTs over the wire; the trap server exposes tracking data. Configure TLS termination or document the reverse proxy requirement in the deploy script.
4. **Session-detached ORM objects** — Throughout the codebase, `async for session in get_session()` yields a session that closes at the end of the block. Objects loaded inside are then used outside the block (e.g., `tracked_users` list in `anomaly_detector.py`, `ml_scorer.py`, `online_tracker.py`). With `expire_on_commit=False` this works for simple attributes but will fail on lazy-loaded relationships.
5. **Unbound `dashboard_task` in shutdown** — `main.py` references `dashboard_task` in the `finally` block, but if an exception occurs before `dashboard_task = asyncio.create_task(...)`, the variable is unbound and shutdown will raise `NameError`.
6. **Docker port mismatch** — `docker-compose.yml` exposes port 8080, but the app's `dashboard_port` defaults to 8443. Reconcile port configuration.
7. **Flask async bridge is fragile** — `_store_visit_async` in `flask_server.py` attempts `asyncio.get_event_loop()` inside a thread-run Flask handler. Since Flask runs in a `ThreadPoolExecutor`, there may not be a running loop, and the fallback creates a new loop per request which defeats connection pooling.
8. **CORS wildcard on dashboard** — `allow_origins=["*"]` with `allow_credentials=True` on the FastAPI middleware is both browser-rejected (browsers refuse credential+wildcard) and overly permissive for a private surveillance dashboard.
9. **No database migrations** — The schema uses `create_all()` with no Alembic or equivalent. Adding a column to a model after deployment will silently leave the column absent from the SQLite file.

## Secret key conflation and authentication posture

The `dashboard_secret_key` setting (default: `"change-me-to-a-random-secret"`) serves triple duty. It is the Flask app's `SECRET_KEY`, the HS256 signing key for JWTs in `dashboard/auth.py`, and the literal password compared in the `/api/auth/login` endpoint:

```python
if request.password != settings.dashboard_secret_key:
    raise HTTPException(...)
token = create_access_token(data={"sub": "admin", "role": "admin"})
```

This means: (a) if the user sets a memorable password, it becomes a weak JWT signing key; (b) if someone intercepts a JWT, they know the signing key, which is also the login password; (c) the system starts without complaint with the insecure default. The deploy script auto-generates a random hex token, which partially mitigates this for scripted deployments but not for manual `.env` setups.

The bot-side authorization is a simple numeric ID comparison (`update.effective_user.id == settings.my_telegram_id`), which is appropriate for a private bot.

## Transport security gaps

Neither the Flask trap server (port 5000) nor the FastAPI dashboard (port 8443) configure TLS. Both bind `0.0.0.0`. The dashboard serves JWT bearer tokens over plaintext HTTP. The trap server collects and stores IP addresses, User-Agent strings, and browser fingerprints from visitors. The deploy script creates a systemd service pointing directly at the Python process without an nginx/caddy reverse proxy, meaning production traffic flows unencrypted.

The `CORSMiddleware` on the dashboard is configured with `allow_origins=["*"]` and `allow_credentials=True`. Per the CORS specification, browsers reject this combination — credentials cannot be sent with a wildcard origin. This is likely a development convenience that would break real cross-origin authenticated requests in production.

## Session lifecycle and ORM detachment

The database module provides `get_session()` as an async generator:

```python
async def get_session() -> AsyncGenerator[AsyncSession, None]:
    factory = get_session_factory()
    async with factory() as session:
        yield session
```

Consumers use it as:
```python
async for session in get_session():
    result = await session.execute(select(TrackedUser)...)
    tracked_users = result.scalars().all()
# session is now closed; tracked_users are detached
for user in tracked_users:
    await self._analyze_user(user)  # accesses user.telegram_id, etc.
```

The `expire_on_commit=False` setting preserves loaded scalar attributes after commit, so immediate attribute access works. However, any code path that triggers a lazy-load relationship (e.g., accessing `tracked_user.story_views` without a prior `joinedload`) after the session closes will raise `DetachedInstanceError`. The current code appears to avoid this by not traversing relationships outside sessions, but the pattern invites future breakage as the codebase grows.

## Shutdown orchestration in main.py

The `main()` function assigns `dashboard_task` midway through the try block:

```python
try:
    # ... several steps that can fail ...
    dashboard_task = asyncio.create_task(run_dashboard(settings))
    # ... more steps ...
finally:
    if dashboard_task and not dashboard_task.done():
        dashboard_task.cancel()
```

If the Telethon connection or bot initialization fails before the `dashboard_task` assignment, the `finally` block will raise `NameError: name 'dashboard_task' is not defined`. Initialize it to `None` before the try block.

Similarly, the Flask trap server launched via `loop.run_in_executor(executor, run_trap_server, ...)` has no cleanup path — `executor.shutdown(wait=False)` is called, but the Flask thread's `app.run()` has no shutdown signal and will linger until the process exits.

## Docker and port configuration mismatch

The Dockerfile exposes ports `5000 8080`. The `docker-compose.yml` maps `8080:8080` for the dashboard. But the application's default `dashboard_port` is 8443, and the systemd service in `deploy.sh` uses the settings defaults (8443). This means Docker Compose will route traffic to a port nothing is listening on unless the user manually sets `DASHBOARD_PORT=8080` in the `.env` file. The healthcheck similarly hits port 8080.

## Flask/async bridge in trapnet

The Flask trap server runs in a `ThreadPoolExecutor` thread. When a visit arrives, `_store_visit_async` tries to find a running asyncio event loop:

```python
try:
    loop = asyncio.get_event_loop()
    if loop.is_running():
        loop.create_task(_store_visit(link_id, visitor_ip, user_agent, referer))
    else:
        asyncio.run(_store_visit(link_id, visitor_ip, user_agent, referer))
except RuntimeError:
    loop = asyncio.new_event_loop()
    ...
```

In a thread, `asyncio.get_event_loop()` in Python 3.10+ raises `DeprecationWarning` or `RuntimeError` if there's no current event loop. The `create_task` path relies on the main thread's loop being accessible, which is not guaranteed from a worker thread. The fallback of creating a new loop per request defeats SQLAlchemy's async engine connection pooling since each new loop gets its own execution context.

## Intelligence pipeline design

The scoring system uses pre-assigned weights across 8 features, with a `MinMaxScaler` normalization pass when multiple users are scored together. The `_normalize_scores` method rescales scores relative to the current cohort, meaning a user's absolute score changes depending on who else is being tracked — a user alone in the tracking list always gets an extreme score. This is a design choice but could confuse operators who expect stable scores.

The `AnomalyDetector` instantiates a fresh `IsolationForest` per user per run (fitting on that user's 28-day window). With the default `n_estimators=100` and a minimum of 7 data points (daily features), the model is fitting 100 trees on 7-28 samples, which produces unstable predictions. The `contamination=0.1` parameter assumes 10% of daily observations are anomalous, which may over-flag on sparse data.

## Test coverage

Tests cover the pattern engine (4 pattern types with positive/negative cases), the scorer (feature computation and classification), and the online tracker (event detection and duration computation). All 35 tests use mock sessions and fake data objects, exercising the logic without a database or Telegram connection.

Not tested: the Flask trap server request handling, the FastAPI dashboard endpoints, the honeypot bait scheduling, the anomaly detector's IsolationForest integration, the WebSocket broadcast/disconnect handling, the deploy script's systemd integration, the main.py orchestration and shutdown sequence, PDF report generation, and the session-lifecycle patterns described above.

## Scheduler design

The `TaskManager` wraps APScheduler's `AsyncIOScheduler` with sensible defaults: `coalesce=True` prevents job pile-up, `max_instances=1` prevents overlapping runs, and a 60-second `misfire_grace_time` handles scheduler lag. The task intervals are aggressive (30-second story checks, 60-second online checks) which will put sustained pressure on Telegram rate limits. The userbot's 1-second rate limiter helps, but checking all tracked users every 30 seconds with a per-request 1-second sleep means a 50-user list takes 50 seconds per cycle — exceeding the 30-second interval and causing back-pressure.
