# Version channel, method-call fixes, and deep validation test suite

This change fixes 7 method-call mismatches that would crash scheduler tasks, dashboard PDF export, and storage export at runtime; adds a `VersionChannel` class that posts version announcements and system status to a configurable Telegram channel; and introduces 105 new tests covering previously-untested modules (trapnet, storage, dashboard, scheduler, intelligence, bot alert_manager). The version channel is wired into `main.py` to announce on startup and exposed via a `/version` bot command.

**Watch for:** The startup version announcement fires after `run_until_disconnected` is blocked — it cannot execute (confirmed). `VersionChannel` constructs a new `Bot` instance from settings if none is injected, which silently creates a second bot object competing with the application's own (likely).

## High-level view

The method-call fixes are straightforward renames (`check_all_tracked_users` → `check_all_targets`, `detect_anomalies` → `run`, `analyze_all_users` → `deep_analysis`, `generate_pdf` → `generate_user_report`) plus a new `rebuild_all_profiles()` method on `BehaviorProfiler`. These were all runtime `AttributeError` crashes waiting to happen.

The version channel is a self-contained module with three posting methods, each guarded by a None-check on `version_channel_id` so the feature is dormant unless configured. However, the startup integration in `main.py` places the announcement call after `userbot.run_until_disconnected()`, which blocks the event loop forever — the announcement will never execute during normal operation.

The deep validation test suite is broad but shallow — 105 tests that verify imports, instantiation, and single-call return shapes. The untested gaps are more interesting than what's covered.

<details>
<summary>Issues (4)</summary>

1. **Startup announcement unreachable** — `version_channel.post_version_update()` is placed after `await userbot.run_until_disconnected()` which blocks indefinitely. Move it before that call so the announcement actually fires on startup.
2. **Duplicate Bot instance on fallback path** — If `VersionChannel` is constructed without an explicit bot, `self.bot` creates a standalone `Bot(token=...)` that doesn't share the application's session pool or updater. In the startup code this is avoided (bot_app.bot is passed), but any future call to `VersionChannel()` without arguments would create a second connection.
3. **`datetime.utcnow()` deprecation** — All new code uses `datetime.utcnow()` which is deprecated since Python 3.12 and returns a naive datetime. Not blocking for Python 3.11 but will break on upgrade.
4. **Version string not sourced from package metadata** — `app_version` is a config field defaulting to `"2.0.0"` with no mechanism to keep it in sync with the CHANGELOG or a `__version__` variable. Easy to drift.

## Startup announcement placement in main.py

The version channel announcement is placed after `await userbot.run_until_disconnected()`. That Telethon call blocks the event loop until the userbot disconnects — which under normal operation means "forever." The announcement will only fire during shutdown (if the userbot disconnects due to an error), not on startup as intended.

```python
await userbot.run_until_disconnected()  # blocks here

# This code is unreachable during normal operation:
try:
    version_channel = VersionChannel(bot=bot_app.bot)
    await version_channel.post_version_update(...)
```

The fix is to move the version channel block above `run_until_disconnected`. The `send_startup_notification` call demonstrates the correct placement pattern — it fires before the blocking call.

## VersionChannel fallback Bot construction

When `VersionChannel(bot=None)` is used, the `bot` property lazily creates `Bot(token=self._settings.bot_token)`. This is a raw `telegram.Bot` instance — no connection pool sharing, no rate-limit coordination with the main application's `Updater`. The fallback path exists as a trap for any caller that instantiates `VersionChannel()` without passing the application bot.

The class would be safer if it required a bot instance (removing the None default) or accepted the `Application` object and extracted `.bot` from it.

## Test coverage gaps

Not tested: the `VersionChannel` startup path in `main.py` (unreachable code — can't be tested as-is), `rebuild_all_profiles` with actual DB records (only import validation), error propagation in scheduler tasks beyond import-time validation, concurrent access to `InMemoryCache`, and the `archive_old_data` deletion path in `DataExporter`.
