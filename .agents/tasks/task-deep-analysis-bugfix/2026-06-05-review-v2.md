# IPC reliability fixes and snake_case/camelCase serialization bridge (v2)

The three blocking bugs from v1 are resolved. The `toCamelCase` regex now matches `/_([a-z0-9])/g`, correctly converting `requires_2fa` → `requires2fa`. The `restart()` method waits for the old process's close event before spawning a new one, eliminating the concurrent-process race. The readiness gate now awaits a promise instead of throwing, so early RPC calls queue until the Python process emits its first stdout byte.

**Watch for:** Credentials (`api_id`, `api_hash`) are persisted to `~/.teleexport/config/settings.json` in plaintext alongside UI preferences (confirmed). Export tasks have no shutdown-time cancellation hook — unfinished tasks produce "Task exception was never retrieved" warnings on process exit (possible).

## High-level view

With the three v1 blockers fixed, the IPC bridge is functionally correct: snake_case/camelCase conversion handles the full key space including digit-after-underscore patterns, the process lifecycle doesn't race, and the readiness gate is non-blocking. The remaining concerns are non-blocking but worth addressing before production: credential isolation from the general settings file, and graceful task teardown on shutdown.

## Credential persistence in settings.json

The `auth_send_code` handler writes API credentials directly into the shared settings dict:

```python
_settings["api_id"] = api_id
_settings["api_hash"] = api_hash
_save_settings(_settings)
```

This persists them to `~/.teleexport/config/settings.json` — the same file that stores theme, language, and output directory. Telegram API ID/hash are per-app credentials (less sensitive than session tokens), but mixing them with user preferences means any code that reads or logs settings inadvertently exposes them. Electron's `safeStorage.encryptString()` would isolate them without requiring OS keychain integration.

## Export task lifecycle on shutdown

The export handler stores `(engine, task)` in `app.exports` and the `run_export` coroutine cleans itself up in `finally`. During normal operation this works. The gap: when the Python process receives SIGTERM (from the bridge's `stop()`), the asyncio event loop cancels all pending tasks. The `finally` block runs and pops the dict entry, but if the task had a pending exception at the moment of cancellation, Python emits a "Task exception was never retrieved" warning to stderr. Adding a `task.add_done_callback` that checks for and logs exceptions — or registering an `atexit`/signal handler that explicitly cancels and awaits all tracked tasks — would silence this.
