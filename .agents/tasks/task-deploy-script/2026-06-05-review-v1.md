# Headless server deployment for TeleExport

A deploy.sh script automates installing Python 3.11 via deadsnakes, pip dependencies, API credential entry, and systemd service registration on Ubuntu servers. A companion run_server.py replaces the Electron IPC loop with a standalone asyncio daemon that handles Telethon auth interactively on first run, then holds the connection as a long-lived service.

Watch for:
- **Credentials written to disk as world-readable plaintext** (confirmed) — settings.json has no `chmod 600`.
- **run_server.py is a keep-alive loop with no work interface** (confirmed) — the daemon stays connected but exposes no way to trigger exports.
- **pip installs run as root with no --user or venv isolation** (confirmed) — systemwide pip contamination risk on shared servers.

## High-level view

The deploy script is comprehensive for a single-user AWS instance: it gates on RAM and disk, installs Python from deadsnakes, validates credentials, writes config, and registers a systemd unit that starts the runner under the invoking user. The auth-before-start pattern (interactive first run, then `systemctl start`) is the right call for Telegram sessions that need a one-time phone code.

run_server.py is a thin glue layer: load config, init TeleExportClient, authenticate if needed, then park on an asyncio Event until SIGTERM. The interface with the existing backend is correct — it reuses TeleExportClient.init/connect/disconnect and AuthManager's send_code/sign_in/sign_in_2fa flow without modification. However, the daemon does nothing after connecting; there's no export trigger mechanism, making the service a perpetual no-op until a future iteration adds a command interface or scheduled export.

The systemd unit is correctly configured for a non-root daemon (User/Group set, WorkingDirectory, journal logging, restart on failure with 10s backoff). The gap is credential file permissions and the absence of environment isolation (venv).

<summary>Issues (7)</summary>

1. **Plaintext credentials without restricted permissions** — `settings.json` containing `api_hash` is created without `chmod 600`. Any user on the server can read it. Add `chmod 600 "${CONFIG_DIR}/settings.json"` after writing.
2. **Systemwide pip install as root** — `python3.11 -m pip install` runs as root without `--user` or a virtualenv, polluting the system Python and risking conflicts. Use a venv at `~/.teleexport/venv/` and point the systemd ExecStart at its interpreter.
3. **Daemon has no export interface** — run_server.py holds a Telethon connection alive but exposes no mechanism (socket, HTTP, signal-based) to request an export. The service is currently a no-op after authentication.
4. **No session-file permission check** — Telethon `.session` files in `~/.teleexport/sessions/` are SQLite databases containing auth keys. The deploy script doesn't restrict their permissions either.
5. **api_hash validation regex too permissive** — The regex `^[a-fA-F0-9]{24,64}$` accepts a wide range when Telegram api_hash values are consistently 32 hex characters. The range might accept garbage inputs that fail at runtime.
6. **No dependency pinning or lockfile** — pip installs use minimum-version specifiers (`>=`). Two deployments a week apart may get different package versions. No mechanism to reproduce a known-good install.
7. **device_model says "Desktop" in headless mode** — TeleExportClient hardcodes `device_model="TeleExport Desktop"`. Telegram may display "Desktop" in active sessions, which is cosmetically misleading for a server deployment but not functionally broken.

## Credential handling and file permissions

The deploy script writes `api_hash` in plaintext to `~/.teleexport/config/settings.json` and calls `chown` to fix ownership, but never restricts the file mode. On a multi-user server (or if a different service is compromised), any process with read access to the user's home directory can extract the API credentials. The fix is one line after the `chown`:

```bash
chmod 600 "${CONFIG_DIR}/settings.json"
```

The same concern applies to the Telethon session file (`~/.teleexport/sessions/server.session`), which stores the auth key. The deploy script creates the directory but doesn't set `chmod 700` on it.

## System-level pip without isolation

Lines 114-123 of deploy.sh run `python3.11 -m pip install` under root (the script requires sudo). This installs packages globally into the system Python 3.11 site-packages. On an AWS instance running only TeleExport this may be tolerable, but on any shared infrastructure it risks version conflicts with other Python services. The conventional pattern for systemd-managed Python services is:

```bash
sudo -u "$REAL_USER" python3.11 -m venv "${REAL_HOME}/.teleexport/venv"
sudo -u "$REAL_USER" "${REAL_HOME}/.teleexport/venv/bin/pip" install ...
```

Then the systemd ExecStart points to the venv interpreter. This also avoids needing root for pip entirely.

## The daemon's empty run loop

```
┌─────────────┐         ┌──────────────┐
│ run_server  │──init──▶│TeleExportClient│
│   .py       │◀─conn───│  (Telethon)    │
└─────┬───────┘         └──────────────┘
      │
      ▼
 asyncio.Event.wait()  ← no work happens here
      │
  SIGTERM ──▶ disconnect
```

After authentication, `run_daemon()` parks on `stop_event.wait()` indefinitely. The Telethon connection stays alive (keepalive pings), but there's no command socket, HTTP API, file-watcher, or scheduled task to actually invoke exports. The existing IPC server (`python/ipc/server.py`) reads JSON-RPC from stdin — which is `/dev/null` under systemd. So the service currently cannot do anything useful post-auth. This needs a follow-up: either expose a Unix socket / HTTP endpoint, or add a cron-like scheduler inside the daemon.

## Auth error propagation

If `sign_in` raises an exception other than the 2FA case (e.g., `PhoneCodeExpired`, `PhoneCodeInvalid`), it propagates to the outer `try/except Exception` in `__main__` and prints "Fatal error: ..." with no guidance. The user won't know they should simply re-run to get a new code. Catching known Telethon exceptions in `interactive_auth` and printing actionable recovery steps would improve the first-run experience.

## Deploy script robustness gaps

The `set -e` combined with `trap cleanup EXIT` handles most failure paths, but:

- `add-apt-repository -y ppa:deadsnakes/ppa` redirects stderr to `/dev/null`. If the PPA is unreachable (e.g., behind a proxy), the script exits with no useful diagnostic.
- The cleanup trap prints a generic "Deployment failed" message with no indication of *which* step failed. Adding `set -o pipefail` and a line-number trap (`trap 'error "Failed at line $LINENO"' ERR`) would improve debuggability.
- The script hardcodes `python3.11` everywhere. Since `pyproject.toml` requires `>=3.11`, the script could detect and accept 3.12+ as a fallback.

## my.telegram.org rate-limiting

The instructions don't mention that my.telegram.org rate-limits login attempts and may block access for 24h after too many failed SMS codes — a common first-time pitfall that leads to support requests.
